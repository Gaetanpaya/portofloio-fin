DÉPLOIEMENT ACTIVE DIRECTORY (AD DS)

Documentation technique détaillant l'installation, la structuration et la sécurisation du contrôleur de domaine sous Windows Server dans le cadre du projet CUB.

1. Prérequis & Préparation du Serveur

Avant de promouvoir un serveur en tant que Contrôleur de Domaine, il est impératif de préparer l'environnement réseau et système pour éviter tout conflit futur.

Actions préliminaires :

  • Nommage : Renommer le serveur avec un nom clair (ex: SRV-AD-CUB) via les paramètres système.
  • Adressage IP fixe : Configurer une adresse IP statique sur la carte réseau (ex: 192.168.10.10).
  • Configuration DNS : Faire pointer le serveur DNS primaire sur l'adresse de bouclage (127.0.0.1) ou sur lui-même, car l'AD a besoin de son propre service DNS pour fonctionner.

2. Installation et Promotion du Rôle AD DS

Le déploiement se fait via le Gestionnaire de Serveur (Server Manager).

Étape 1 : Ajout des rôles

Dans le Gestionnaire de Serveur, cliquer sur Ajouter des rôles et des fonctionnalités. Sélectionner Services AD DS (Active Directory Domain Services). Le système cochera automatiquement l'ajout du rôle DNS, indispensable à l'AD.

Étape 2 : Promotion en Contrôleur de Domaine

Une fois l'installation terminée, cliquer sur le drapeau d'alerte jaune et sélectionner "Promouvoir ce serveur en contrôleur de domaine".

  • Choisir "Ajouter une nouvelle forêt" (puisqu'il s'agit du premier serveur).
  • Nom du domaine racine : cub.local (nom de domaine interne).
  • Définir le mot de passe DSRM (Mode de restauration des services d'annuaire), crucial en cas de crash de la base de données AD.
Vérification : Après le redémarrage automatique, le serveur affiche le nom du domaine à l'écran de connexion (ex: CUB\Administrateur), confirmant la réussite de la promotion.

3. Structuration de l'Annuaire (OU)

Pour ne pas mélanger tous les utilisateurs et ordinateurs dans les dossiers par défaut, il faut utiliser l'outil Utilisateurs et ordinateurs Active Directory (dsa.msc) pour créer une hiérarchie.

Création des Unités d'Organisation (OU)

J'ai créé des Unités d'Organisation (OU) pour refléter les différents services de l'entreprise. Cela permet de déléguer l'administration et d'appliquer des stratégies (GPO) ciblées.

CUB.local
 ├── OU=Direction
 │    ├── Utilisateurs
 │    └── Ordinateurs
 ├── OU=Informatique
 │    ├── Utilisateurs
 │    └── Ordinateurs
 └── OU=Ressources_Humaines

Gestion des Droits (Bonnes pratiques)

Les permissions sur les dossiers partagés de l'entreprise ne sont jamais données directement à un utilisateur. J'ai créé des Groupes de Sécurité Globaux (ex: GG_Lecture_RH) auxquels j'attribue les droits NTFS, puis j'ajoute les utilisateurs dans ces groupes.

4. Sécurisation via les Stratégies de Groupe (GPO)

Les GPO (Group Policy Objects) permettent d'automatiser la configuration des postes clients et de verrouiller l'environnement de travail pour des raisons de sécurité.

Exemples de GPO déployées dans le contexte CUB :

  • Mappage de lecteur réseau : Connexion automatique du lecteur S: (dossier partagé commun) à l'ouverture de session de l'utilisateur.
  • Sécurité des postes : Désactivation de l'accès au Panneau de configuration et à l'Invite de commandes (cmd) pour les utilisateurs standards (OU=Ressources_Humaines).
  • Stratégie de mots de passe : Obligation d'avoir des mots de passe complexes (min. 12 caractères) et renouvellement tous les 90 jours (GPO "Default Domain Policy").
Commande utile : Sur un poste client (Windows 10/11) intégré au domaine, la commande gpupdate /force permet de forcer l'application immédiate des nouvelles GPO sans avoir à redémarrer.
